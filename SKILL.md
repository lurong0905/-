# 发货通知资料生成

## 用途

使用本流程，把合同文件、发货信息搜集表、客户发货模板、Skill 内置客户默认信息和仓库装箱清单整理成一套可提交的发货资料。

只要存在客户模板，就优先复制并编辑模板，保持原有版式、表格、勾选框、页眉页脚和客户固定措辞，不要重新手工仿制表单。

对重复发货的合同，优先使用合同文件夹内的 `发货信息搜集表.xlsx` 作为小型权威输入。PDF 合同只在首次建表、信息表缺失字段需要补查，或用户明确要求重新预填时读取；正式生成文件时不要默认把大 PDF 再读一遍。

## 工作流程

1. **确认工作目录和信息搜集表**
   - 查找客户模板文件夹，常见名称为 `发货通知模版` 或类似名称。
   - 列出合同文件、`发货信息搜集表.xlsx`、仓库装箱清单、已有输出文件夹。
   - 忽略名称以 `~$` 开头的 Office 临时锁定文件。
   - 如果合同文件夹没有 `发货信息搜集表.xlsx`，先运行 `scripts/create_shipping_info_table.py` 自动预填一份表，再让用户核对或补齐空缺字段。
   - 如果已经有 `发货信息搜集表.xlsx`，先读取该表；不要默认重新读取 PDF 合同。

2. **识别客户体系**
   - 优先从 `发货信息搜集表.xlsx` 的 `合同固定信息` 读取客户体系、买方、合同号和工厂代码。
   - 如果信息表缺失，再根据文件名、买方名称、合同编号、模板标题判断属于 `中核`、`国核`、`广核` 或其他客户。
   - 识别为 `国核` 后还要继续区分具体电厂或采购方；`国核示范电站有限责任公司` 与 `山东核电有限公司` 的到货通知、唛头、装箱单模板不同，不要只按“国核”共用同一套模板。
   - 对于已知客户体系，生成文件前先读取 `references/customer-template-rules.md`；创建或解释信息表时读取 `references/shipping-info-table.md`。
   - 如果客户体系不明确，先根据买方、合同编号格式、模板标题和用户要求的输出文件进行推断；只有当判断会影响文件清单或必填字段时，才向用户确认。

3. **提取权威信息**
   - 信息优先级为：用户本次明确输入 > `发货信息搜集表.xlsx` > 仓库装箱清单 > Skill 内置客户默认信息 > PDF 首次预填结果。
   - 从 `合同固定信息` 读取：合同名称、合同号或订单号、买方、卖方、交货地点、交货时间、收货单位和联系人、合同管理联系人、发货单位联系人等一次性信息。
   - 从 `批次发货信息` 读取：批次、发货清单文件、计划发运时间、预计到达时间、运单号或快递单号、运输方式、装卸货方式、储存级别及条件等每批信息。
   - 从仓库装箱清单中提取：物项行、合同序号或订单行号、物资编码、名称、规格型号、数量、单位、箱号、尺寸、重量、包装方式、储存等级、制造商、质保等级、生产日期、有效期、随箱资料。
   - 从本 Skill 的参考规则中提取客户默认收货地址、常用模板字段、客户特殊要求；从用户补充资料中提取项目专属联系人、发货单位信息和签署信息。
   - 除非用户另有说明，仓库装箱清单是实际发货物项和箱级数据的优先依据。
   - 遇到参考规则中标注为“需用户选择”的字段，或信息表中对应字段为空时，必须在生成最终文件前向用户确认选项，不要自行默认勾选。
   - `发运时间` 不能从合同资料、模板、装箱清单或当前日期中推断；生成最终文件前必须从信息表或用户回复中取得明确值。不同模板中的 `计划发运时间`、`发货日期` 按此规则处理。

4. **生成成套文件**
   - 将对应客户模板复制到新的输出文件夹，输出文件夹可按合同、批次或日期命名。
   - 生成中核 `发货通知.docx` 时，可使用 `scripts/generate_cnnp_shipping_notice.py` 从 `发货信息搜集表.xlsx`、仓库发货清单和中核模板生成待确认版。
   - 生成中核 `唛头（每箱1张）.docx` 时，可使用 `scripts/generate_cnnp_shipping_marks.py` 从 `发货信息搜集表.xlsx`、仓库发货清单和中核唛头模板生成每箱一页的唛头。
   - 生成中核 `装箱清单（每箱1张）.docx` 时，可使用 `scripts/generate_cnnp_packing_lists.py` 只为达到“单箱不同物品类别 3 类及以上”的箱号生成装箱清单。
   - 生成广核 `国内到货通知.docx`、广核 `唛头.docx`、国核 `到货通知单`、国核 `唛头`、国核 `装箱单` 时，可使用 `scripts/generate_cgn_snp_shipping_docs.py` 生成待确认版。
   - 生成国核装箱单时要按客户原模板的页块填写，保持一页一个装箱单；不要把模板表格重排成多个重叠表，也不要为了塞进页面而全局压成过小字号。
   - 先填写发货通知或到货通知，再填写箱级装箱单或装箱清单，最后生成每箱唛头。
   - 如果客户模板或合同要求质量符合声明、自检报告、随箱文件清单等附件，一并生成或保留待补字段。
   - 保持客户模板的原有措辞、勾选框、表格结构、页面布局、合并单元格和批次命名风格。
   - 批次名称沿用模板习惯，例如 `第一批`、`第二批`、`第三批`。

5. **完成前核对**
   - 核对每个发货物项是否同时出现在通知表和装箱单或装箱清单中。
   - 核对物项项数、总数量、总箱数、箱号范围、尺寸、重量、包装方式、储存等级、随箱资料名称。
   - 核对通知日期、计划发运时间、预计到达时间、收货联系人、交货地点、发货联系人是否在所有文件中一致。
   - 清理或明确标注仍未确认的模板占位符。
   - 对 Word 模板类输出至少做两层验证：读取文档结构核对表格、箱号和关键字段；再用 Word 实际分页或导出 PDF 抽查，确认无重叠、无空白尾页、无跨页挤压。
   - 如果 PDF 是扫描件且没有文字层，能使用 OCR 时先识别；无法识别时，在最终说明中明确哪些合同条款未能从该 PDF 中完整抽取。

## 输出要求

用户要求生成文件时，必须在当前工作目录中生成实际 Word 或 Excel 交付文件，不要只在对话中总结。

最终回复中列出已生成文件，说明缺失信息、处理假设，以及需要用户确认的未完成字段或占位符。

## 文件处理要求

处理 Word、Excel、PDF 文件时使用对应的文档或表格读取方式。编辑 Word 和 Excel 时优先修改模板副本，避免破坏原模板。

处理中文路径时，尽量显式使用 UTF-8，并从当前工作目录解析路径，减少中文路径编码问题。
